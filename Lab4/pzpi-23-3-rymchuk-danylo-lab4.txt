2 МЕТА РОБОТИ
	Мета роботи: продемонструвати можливість горизонтального масштабування бекенда системи HobbyRent для роботи з великим навантаженням, провести навантажувальне тестування та проаналізувати залежність пропускної здатності системи від кількості серверних інстансів.
У межах роботи необхідно:
    • обрати стратегію масштабування (горизонтальне/вертикальне);
    • реалізувати технічне рішення для масштабування (Docker Compose + Nginx Load Balancer);
    • налаштувати декілька інстансів сервера зі спільною базою даних;
    • провести навантажувальне тестування за допомогою інструменту Locust;
    • проаналізувати результати тестування та виявити вузькі місця;
    • завантажити конфігурації на GitHub.


	

	 



3 ОПИС ВИКОНАНОЇ РОБОТИ
	3.1 Стратегія масштабування
Для системи HobbyRent обрано горизонтальне масштабування, яке передбачає запуск декількох ідентичних інстансів сервера, що виконують однакові функції для різних користувачів. Такий підхід має низку переваг порівняно з вертикальним масштабуванням:
    1. Лінійне зростання продуктивності: додавання нового інстансу сервера пропорційно збільшує загальну пропускну здатність системи.
    2. Відмовостійкість: вихід з ладу одного інстансу не призводить до повної недоступності системи — Nginx автоматично перенаправляє трафік на інші працездатні сервери.
    3. Гнучкість: кількість інстансів може динамічно змінюватися залежно від навантаження.
    4. Економічна ефективність: дешевше додати декілька середніх серверів, ніж один потужний.
3.2 Технічні рішення
3.2.1 Архітектура розгортання:
Система розгорнута у контейнерах Docker за допомогою Docker Compose. Архітектура включає такі компоненти:
    1. Nginx Load Balancer — вхідна точка системи, приймає всі HTTP-запити від клієнтів та розподіляє їх між серверними інстансами за алгоритмом round-robin (циклічне перемикання).
    2. 3 інстанси Spring Boot сервера (server1, server2, server3) — кожен інстанс є повною копією бекенду, що виконує однакову бізнес-логіку. Усі інстанси підключені до спільної бази даних PostgreSQL.
    3. PostgreSQL — єдина база даних, до якої підключаються всі серверні інстанси. Це забезпечує консистентність даних: зміни, зроблені через один інстанс, одразу доступні через інші.
3.2.2 Конфігурація Nginx:
upstream backend_servers {
    server server1:8080;
    server server2:8080;
    server server3:8080;
}

server {
    listen 80;
    location / {
        proxy_pass http://backend_servers;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
    }
}
3.2.3 Конфігурація Docker Compose:
Кожен інстанс сервера отримує однакові змінні оточення для підключення до бази даних:
environment:
  - SPRING_DATASOURCE_URL=jdbc:postgresql://postgres:5432/hobbyrent_db
  - SPRING_DATASOURCE_USERNAME=postgres
  - SPRING_DATASOURCE_PASSWORD=postgres123


3.3 Аналіз вузьких місць
При горизонтальному масштабуванні бекенда вузькими місцями можуть стати:
    1. База даних (PostgreSQL):
        ◦ Усі інстанси сервера звертаються до однієї БД, тому при великому навантаженні може виникнути конкуренція за ресурси БД (CPU, I/O, connection pool).
        ◦ Рішення: налаштувати connection pooling (HikariCP), оптимізувати запити, використовувати індекси.
    2. Мережева пропускна здатність:
        ◦ Nginx повинен обробляти всі вхідні запити, тому може стати вузьким місцем при дуже високому навантаженні.
        ◦ Рішення: використати декілька Nginx інстансів з власним load balancer (наприклад, HAProxy).
    3. Стан сесій:
        ◦ Якщо сервер зберігає стан сесії в пам'яті (in-memory), то при балансуванні навантаження користувач може потрапити на інший інстанс, де його сесія відсутня.
        ◦ Рішення: використовувати stateless архітектуру (JWT токени) або централізоване сховище сесій (Redis).
    4. Файлова система:
        ◦ Якщо сервер завантажує файли на локальний диск, вони не будуть доступні на інших інстансах.
        ◦ Рішення: використовувати централізоване файлове сховище (S3, NFS).
У системі HobbyRent вузьким місцем є база даних, оскільки всі операції читання/запису проходять через неї. Однак для навчальних цілей та помірних навантажень PostgreSQL справляється з навантаженням від 3 інстансів сервера.
	3.4 Навантажувальне тестування
Для проведення навантажувального тестування обрано інструмент Locust (Python), який дозволяє моделювати різні сценарії використання системи та збирати детальну статистику.
3.4.1 Сценарії тестування
    1. Звичайний користувач (HobbyRentUser):
    • Перегляд списку спорядження (5 запитів)
    • Отримання інформації про користувача (2 запити)
    • Створення нового спорядження (1 запит)
3.4.2 Параметри тестування:
    • Кількість віртуальних користувачів: 100
    • Швидкість створення користувачів: 10 users/second
    • Тривалість кожного тесту: 3 хвилини

3.4.4 Аналіз результатів
Результати тестування показали, що система демонструє стабільну продуктивність при збільшенні кількості серверних інстансів. Пропускна здатність залишається на рівні ~49-50 запитів на секунду, а середній час відповіді не перевищує 14 ms.
Відсутність лінійного масштабування пояснюється такими факторами:
    1. База даних як вузьке місце: Усі інстанси сервера використовують спільну базу даних PostgreSQL, що створює конкуренцію за ресурси (CPU, дисковий I/O, пул з'єднань). При збільшенні кількості серверів навантаження на БД зростає, що обмежує загальну продуктивність системи.
    2. Обмежене тестове навантаження: Оскільки параметри навантажувального тесту (100 користувачів) були однаковими для всіх конфігурацій, загальна кількість запитів залишалася приблизно однаковою, просто розподіляючись між більшою кількістю серверних інстансів.
    3. Накладні витрати на балансування: Nginx Load Balancer додає невелику затримку на маршрутизацію запитів між серверами.
4 ВИСНОВКИ
	У результаті виконання лабораторної роботи було реалізовано горизонтальне масштабування бекенда системи HobbyRent за допомогою Docker Compose та Nginx Load Balancer. Розгорнуто 3 ідентичні інстанси Spring Boot сервера зі спільною базою даних PostgreSQL, що дозволило забезпечити стабільну роботу системи.
Проведено навантажувальне тестування за допомогою інструменту Locust, яке показало:
    • Стабільну пропускну здатність на рівні ~49-50 запитів на секунду
    • Низький середній час відповіді (13-14 ms)
    • Повну відсутність помилок (0% failure rate)
Виявлено, що основним вузьким місцем системи є база даних PostgreSQL, оскільки всі інстанси сервера звертаються до неї. Для подальшого масштабування необхідно оптимізувати запити до БД, налаштувати connection pooling та розглянути можливість реплікації бази даних.
Отримані результати підтверджують, що обрана стратегія горизонтального масштабування забезпечує високу доступність та стабільність системи, що є критично важливим для веб-застосунків з великою кількістю користувачів.
