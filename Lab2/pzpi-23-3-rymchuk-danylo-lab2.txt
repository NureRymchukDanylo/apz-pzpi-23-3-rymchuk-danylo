2 МЕТА РОБОТИ
     	Мета роботи: розробити мобільний програмний застосунок для платформи Android, який є частиною системи HobbyRent та забезпечує взаємодію користувача з серверною частиною системи для перегляду каталогу спорядження, оформлення оренди та моніторингу IoT-даних.
     У межах роботи необхідно:
* розробити мобільний застосунок з використанням Kotlin та Jetpack Compose;
* забезпечити взаємодію з REST API бекенду через Retrofit;
* реалізувати архітектуру MVVM для розділення логіки та інтерфейсу;
* створити необхідні UML діаграми (Use Case, Component, Activity, State);
* завантажити код на GitHub.
     
     
     	

	 




3 ОПИС ВИКОНАНОЇ РОБОТИ
	3.1 Інженерні рішення та архітектура
     Для розробки мобільного застосунку обрано платформу Android з використанням мови програмування Kotlin. Вибір обумовлений широкою поширеністю платформи, сучасними інструментами розробки та інтеграцією з екосистемою Google.
     Технологічний стек:
* UI Framework: Jetpack Compose - сучасний декларативний фреймворк для побудови інтерфейсу, що дозволяє писати UI як функції стану.
* Архітектура: MVVM (Model-View-ViewModel) - забезпечує чітке розділення відповідальності між інтерфейсом, бізнес-логікою та даними.
* Мережевий шар: Retrofit 2.9.0 - типобезпечний HTTP-клієнт для взаємодії з REST API, з використанням Gson Converter для автоматичної десеріалізації JSON.
* Асинхронність: Kotlin Coroutines - для виконання мережевих запитів у фоновому потоці без блокування UI.
* Управління станом: StateFlow/MutableState - реактивне оновлення інтерфейсу при зміні даних.

     3.2 Взаємодія з іншими частинами системи
     Мобільний застосунок взаємодіє з серверною частиною (Spring Boot) через REST API. Оскільки сервер розгорнуто локально на комп'ютері розробника, для емулятора Android використовується спеціальна адреса 10.0.2.2, яка є alias для localhost хост-машини.
     Основні ендпоінти:
* GET /api/equipment - отримання списку всього спорядження
* POST /api/rentals/start - оформлення оренди (параметри: equipmentId, userId, endIsoDate)

     3.3 Основні компоненти застосунку
     Data Layer:
     * Equipment, User, Rental - data-класи, що відповідають сутностям серверної бази даних
     * ApiService - інтерфейс Retrofit з описом ендпоінтів
     * RetrofitClient - синглтон для створення экземпляра Retrofit
     * EquipmentRepository - репозиторій для отримання даних з API
		ViewModel Layer:
     * EquipmentViewModel - управляє станом додатку, завантажує дані через Repository, надає їх для UI через State
     UI Layer (Jetpack Compose):
     * MainActivity - головна активність з навігацією та Bottom Navigation Bar
     * CatalogScreen - екран каталогу зі списком спорядження та діалогом оренди
     * MonitoringScreen - екран IoT-моніторингу орендованого спорядження

3.4 Функціональні можливості
Екран каталогу:
    * Відображення списку доступного спорядження у вигляді карток
    * Фільтрація за статусом (AVAILABLE/RENTED)
    * Діалогове вікно для вибору кількості годин оренди
    * Автоматичний розрахунок вартості
    * Оновлення списку після успішної оренди
Екран моніторингу:
    * Відображення тільки орендованого спорядження (статус RENTED)
    * Показ IoT-даних: рівень заряду батареї, температура, GPS-координати
    * LIVE-індикатор активного моніторингу
    * Порожній стан з дружнім повідомленням, якщо немає активних оренд
Навігація:
    * Bottom Navigation Bar для перемикання між екранами
    * Збереження стану при перемиканні вкладок

4 ВИСНОВКИ
     	У результаті виконання лабораторної роботи було реалізовано горизонтальне масштабування бекенда системи HobbyRent за допомогою Docker Compose та Nginx Load Balancer. Розгорнуто 3 ідентичні інстанси Spring Boot сервера зі спільною базою даних PostgreSQL, що дозволило забезпечити стабільну роботу системи.
     Проведено навантажувальне тестування за допомогою інструменту Locust, яке показало:
* Стабільну пропускну здатність на рівні ~49-50 запитів на секунду
* Низький середній час відповіді (13-14 ms)
* Повну відсутність помилок (0% failure rate)
     Виявлено, що основним вузьким місцем системи є база даних PostgreSQL, оскільки всі інстанси сервера звертаються до неї. Для подальшого масштабування необхідно оптимізувати запити до БД, налаштувати connection pooling та розглянути можливість реплікації бази даних.
     Отримані результати підтверджують, що обрана стратегія горизонтального масштабування забезпечує високу доступність та стабільність системи, що є критично важливим для веб-застосунків з великою кількістю користувачів.
