3 ОПИС ВИКОНАНОЇ РОБОТИ
	3.1 Загальні відомості про систему
     Назва системи: Discord
     Розробник: Discord Inc. (заснована у 2015 році Джейсоном Сітроном та Станом Вішневським)
     Призначення системи: Discord - це платформа для голосового, відеового та текстового спілкування, орієнтована на спільноти геймерів, освітні заклади, робочі команди та інші групи користувачів. Система надає можливість створення серверів, текстових та голосових каналів, обміну мультимедійним контентом, інтеграції зі сторонніми сервісами.
     Основні користувачі: понад 560 мільйонів активних користувачів щомісяця.
     Ключові функції:
* Текстові повідомлення в реальному часі
* Голосове та відеозв'язок
* Демонстрація екрана 
* Створення спільнот з гнучкою системою ролей та дозволів
* Інтеграція з іграми та зовнішніми сервісами
* Боти та автоматизація
* Прямі повідомлення та групові чати
     3.2 Архітектурний стиль системи
     Discord використовує гібридну архітектуру, що поєднує кілька архітектурних підходів:
1. Мікросервісна архітектура - основна архітектура backend-частини, де кожен сервіс відповідає за окрему функціональність (аутентифікація, повідомлення, голосовий зв'язок тощо).
2. Подієво-орієнтована архітектура - сервіси спілкуються через події та повідомлення, що забезпечує слабку зв'язність та високу масштабованість.
3. Клієнт-серверна архітектура - клієнти взаємодіють з центральними серверами через API.
4. Real-time архітектура - використання WebSocket для миттєвої доставки повідомлень та оновлень стану.
     Обґрунтування вибору архітектури:
     Початково Discord використовував монолітну архітектуру на основі Django, але зі зростанням кількості користувачів довелося перейти на мікросервіси. Це дозволило:
* Масштабувати окремі компоненти незалежно
* Використовувати різні мови програмування для різних задач (Python, Go, Rust, Elixir)
* Забезпечити високу доступність та відмовостійкість
* Прискорити розробку та впровадження нових функцій
     3.3 Основні компоненти системи
     1. Клієнтський рівень:
* Web Client - React.js веб-додаток
* Desktop Client - Electron-додаток (Windows, macOS, Linux)
* Mobile Client - нативні додатки (iOS - Swift, Android - Kotlin/Java)
     
     2. API Gateway:
* Єдина точка входу для всіх клієнтів
* Маршрутизація запитів до відповідних мікросервісів
* Аутентифікація та авторизація
* Rate limiting та захист від DDoS-атак
     3. Мікросервіси:
* Auth Service - аутентифікація та управління сесіями
* User Service - профілі користувачів, налаштування
* Guild Service - управління серверами 
* Channel Service - текстові та голосові канали
* Message Service - обробка та зберігання повідомлень
* Presence Service - статуси користувачів
* Voice/Video Service - голосовий та відеозв'язок
* Media Service - завантаження та обробка файлів
* Notification Service - push-сповіщення
* Search Service - пошук по повідомленнях
     4. Рівень реального часу:
* WebSocket Service - підтримка постійних з'єднань для миттєвої доставки повідомлень
* Обробка понад 10 мільйонів одночасних WebSocket-з'єднань
     
     5. Рівень зберігання даних:
* Cassandra/ScyllaDB - основна NoSQL база даних для повідомлень та подій
* PostgreSQL - реляційна база для структурованих даних 
* Redis - кешування та управління сесіями
* Elasticsearch - повнотекстовий пошук
* S3-compatible storage - зберігання медіафайлів
     6. Message Brokers:
* RabbitMQ - черги повідомлень між мікросервісами
* Kafka - обробка потоків подій
     7. Інфраструктурні компоненти:
* CDN (Cloudflare, Fastly) - доставка статичного контенту та медіа
* Load Balancers - балансування навантаження
* Docker + Kubernetes - оркестрація контейнерів
* Prometheus + Grafana - моніторинг та метрики
     3.4 Взаємодія компонентів
     Потік даних при надсиланні повідомлення:
1. Користувач надсилає повідомлення через клієнт
2. Клієнт відправляє POST-запит до API Gateway
3. API Gateway перевіряє автентифікацію та пересилає запит до Message Service
4. Message Service зберігає повідомлення у Cassandra
5. Message Service публікує подію "message.created" у RabbitMQ
6. WebSocket Service підписується на події та розсилає повідомлення всім підключеним клієнтам у каналі
7. Notification Service отримує подію та надсилає push-сповіщення офлайн-користувачам
8. Search Service індексує повідомлення в Elasticsearch
     Протоколи взаємодії:
* HTTPS/REST - для API-запитів
* WebSocket - для real-time оновлень
* gRPC - для внутрішньої комунікації між мікросервісами
* UDP/WebRTC - для голосового та відеозв'язку
     3.5 Масштабованість та надійність
     Механізми балансування навантаження:
* Використання Cloudflare для глобального розподілу трафіку
* Kubernetes автоматично масштабує кількість под залежно від навантаження
* Horizontal Pod Autoscaler додає нові інстанси сервісів при зростанні навантаження
     Відмовостійкість:
* Реплікація баз даних 
* Multi-region deployment - сервіси розгорнуті у кількох географічних регіонах
* Circuit Breaker патерн для запобігання каскадним відмовам
* Автоматичне перезапускання впалих контейнерів
     Кешування:
* Redis кешує часто використовувані дані 
* CDN кешує статичні файли 
* Time-to-live стратегії для різних типів даних
     Реплікація:
* Cassandra реплікує дані між вузлами з коефіцієнтом реплікації 3
* PostgreSQL використовує streaming replication для читання
* Redis Sentinel для автоматичного failover
     Контейнеризація та хмарна інфраструктура:
* Всі сервіси упаковані у Docker-контейнери
* Kubernetes оркеструє розгортання, масштабування та управління
* Використання Google Cloud Platform та Amazon Web Services 
* Infrastructure as Code

     Приклад 3: Мікросервіс на Go (спрощений приклад)
      package main
      
      import (
          "encoding/json"
          "net/http"
      )
      
      type Message struct {
          ID        string `json:"id"`
          ChannelID string `json:"channel_id"`
          Content   string `json:"content"`
          AuthorID  string `json:"author_id"`
      }
      
      type MessageService struct {
          db    *Database
          queue *MessageQueue
      }
      
      func (s *MessageService) CreateMessage(w http.ResponseWriter, r *http.Request) {
          var msg Message
          json.NewDecoder(r.Body).Decode(&msg)
          
          // Збереження у базі даних
          s.db.Save(&msg)
          
          // Публікація події у чергу
          s.queue.Publish("message.created", msg)
          
          // Відповідь клієнту
          w.WriteHeader(http.StatusCreated)
          json.NewEncoder(w).Encode(msg)
      }

70 type Message struct {
71     ID        string `json:"id"`
72     ChannelID string `json:"channel_id"`
73     Content   string `json:"content"`
74     AuthorID  string `json:"author_id"`
75     Timestamp string `json:"timestamp"`
76 }

78 type MessageService struct {
79     db    *Database
80     queue *MessageQueue

83 func (s *MessageService) CreateMessage(w http.ResponseWriter, r *http.Request) {
84     var msg Message
85     if err := json.NewDecoder(r.Body).Decode(&msg); err != nil {
86         http.Error(w, err.Error(), http.StatusBadRequest)
87         return
88     }
89  
90     // Збереження у базі даних
91     if err := s.db.Save(&msg); err != nil {
92         http.Error(w, err.Error(), http.StatusInternalServerError)
93         return
94     }
95  
96     // Публікація події у чергу повідомлень
97     if err := s.queue.Publish("message.created", msg); err != nil {
98         log.Printf("Failed to publish event: %v", err)
99     }
100 
101    // Відповідь клієнту
102    w.Header().Set("Content-Type", "application/json")
103    w.WriteHeader(http.StatusCreated)
104    json.NewEncoder(w).Encode(msg)
105 }
